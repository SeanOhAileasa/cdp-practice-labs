Exercise 1 - High Availability and Scalability in a Private Cloud Environment

Scalability and High Availability concepts refer to always ensuring that your infrastructure can satisfy the demand and utilization needed by end users. This is a key focus area when planning an infrastructure, as not performing this thoroughly can result in full denial of service to end users.

High Availability (HA) is a prime motive to migrate to a cloud infrastructure and ensures redundancy, constant monitoring, and failovers are in place to prevent denial of service. Public clouds also implement Availability Zones (AZs) that are isolated, fault-tolerant and highly unlikely to fail simultaneously and can be a great tool when ensuring high availability.

Effectively scaling an infrastructure in a cloud environment comes in different forms and ultimately depends on the specific scenario of what needs to be scaled and will vary depending on the circumstances. These options for scaling include:

  • Auto scaling - This involves adding resources to maintain on-demand availability, typically employed in public cloud environments.
  • Horizontal Scaling - This involves splitting a workload to multiple host machines to make for parallel processing.
  • Vertical Scaling - This involves changing resources to handle appropriate performance demands, such as purchasing more hardware to implement in a private environment or incorporating more resources in a public environment.
  • Cloud Bursting - This involves overflowing traffic directed to a cloud infrastructure when existing infrastructure and resources hit 100% utilization, “bursting” your own infrastructure and “flowing” it to a second cloud infrastructure.

It is key to consider different applications, Virtual Machines (VMs), containers (packages of software, code, and configurations in a single location), clusters (a collection of virtual machine nodes connected via a Virtual Private Cloud) and other types of resources when considering which scaling options fit them best. Typically, it is recommended that these are configured to be Auto Scaling, scaling when they need to.

To ensure HA may require spreading applications over different AZs, VMs have been configured to allow failovers, and containers are hosted inside a Swarm.

Task 1 - Private Cloud Scalability and High Availability Options

Many Hypervisors allow both manual and dynamic scaling of Virtual Machines (VMs) in a multitude of ways, provided that the host device has the partition capacity to enable these changes.

This is invaluable in a private cloud infrastructure and will likely need to be amended to ensure integrity and high availability.

In this task, we will explore scalability controls in Hyper-V.

Step 1

Connect to PLABDC01.

The Server Manager window will open by default.

Step 2

At the top of the page, navigate to and select Tools > Hyper-V Manager.

Step 3

This will open the Hyper-V Manager window.

Notice a pre-configured device already exists as part of this infrastructure, named PLABWIN101, which will appear under the Virtual Machines section.

Under Virtual Machines, right-click PLABWIN101, and click Settings….

Note: In a real-world scenario, it is likely that multiple VMs may be configured on a single device using Affinity model.

Step 4

This will open the configuration settings for the PLABWIN101 device.

Under the Hardware section in the left-hand pane, click Memory.

Note: You will notice that, by default, this device has been configured with 512MB RAM, which we can increase to improve the performance of this device.

You can increase this directly by specifying a new amount of memory in the RAM textbox at the top of the page,

For this demonstration, leave it as 512 MB and tick the Enable Dynamic Memory checkbox.

Step 5

Ticking the Enable Dynamic Memory checkbox will enable the Dynamic Memory option, meaning that this device will dynamically scale depending on the system resource utilization by the host machine.

Note: In a real-world scenario, this is a good Auto-scaling option to use in Anti-Affinity models, as the risk of hitting max host capacity is less, being that there is only one device. Using this option, you can either leave the Maximum RAM to the default entry of 104857MB (which is 1048GB) - effectively meaning it will scale upwards until it hits the max capacity of the host device, or instead manually specify a maximum upper limit. This option can be helpful if you have multiple devices working together on a single machine in an Affinity configuration, although this should be carefully planned to avoid the host becoming a SPOF. The maximum amount supported by Hyper-V is 1,048,576MB, or 1TB.

In the Maximum RAM textbox, type over the existing value with:

2048

Notice you can also specify the weight of priority for this device to have access to memory using the Memory weight setting.

From Memory weight section, click and drag the scale to High.

Click Apply at the foot of the window.

Step 6

Next, under the Hardware section in the left-hand pane, click Processor.

Notice you can specify the number of Virtual Processors (VCPU) that the VM has access to.

Note: This can be helpful to increase for devices that have a higher amount of CPU utilization, and, again, is limited by the number of processors available on the host machine. For example, if a machine was going to be used for specifically high-CPU intensive tasks, then it would be prudent to ensure that it has access to more CPU power than other machines. The maximum limit supported by Hyper-V is 240 VCPUs

In the Number of virtual processors textbox, type over the existing value with:

3

Step 7

Once you make this change, you will notice that the percentages below change depending on the load. This will apply to the host and other VMs running concurrently.

Note: With 3 VCPUs, notice that the Percent of total system resources has increased to 75%. With 4 or more, this would increase to 100% and higher.

Click Apply at the bottom of the page.

Step 8

Finally, under the Management section in the left-hand pane, click Checkpoints.

Note: to ensure high availability and minimal downtime of machines as part of an infrastructure in Hyper-V, it is imperative that checkpoints are created frequently to allow easy recovery and failover if any single device fails. This can be helpful in development environments, as code deployments can sometimes have negative effects on machines and may require frequent rollbacks.

Step 9

Tick the Use automatic checkpoints checkbox.

Note: Ticking the Use automatic checkpoints checkbox will automate the process, creating checkpoints automatically. This can be helpful, although it does remove some direct control if relied on 100% of the time. It is key to ensure that, as well as automatic checkpointing, manual reviews take place, and more checkpoints are done as necessary. 7d0ac9fb-5de8-4057-8699-2545ccd83fb6

Step 10

Click OK at the foot of the page to close the Settings for PLABWIN101 on PLABDC01 window.

Step 11

Close the Hyper-V Manager and Server Manager windows.

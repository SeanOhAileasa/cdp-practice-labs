Exercise 1 - Exploring General Logging in a Public Cloud Environment

Logging is the heart of an infrastructureâ€™s security posture when it comes to ensuring cloud security. Logs provide full visibility to administrators on which resources are currently functional and which may need some remediation. Logs can also provide key insights during an incident response plan and provide potential markers and flags to look for in the future to improve and iterate on a response plan.

For example, an infrastructure could have a device that shows irregularities on security logs before it fully goes down. In the future, if this same device started showing the same irregularities, it could act as a good indicator that another issue was about to happen, further improving incident response procedures.

It is important to remember that, depending on the size of an infrastructure, the sheer volume of data being reported by multiple devices can be staggering, so logging should be done in accordance with what needs to be monitored. Logs should be relevant, easy to access, accurate, and analyzable.

Task 1 - Explore Azure Monitor

Monitor is a tool in Azure that allows real-time analysis and logging of multiple types of logs, including data received by specific resource groups, resource types or locations.

In this task, we will explore Monitor in Azure.

Step 1

Connect to PLABWIN10.

Step 2

From the Taskbar, select Microsoft Edge.

Note: When opening the Microsoft Edge browser, it will redirect you to the Azure portal and sign in automatically. An Azure subscription will be provided for all lab activities that utilize the Azure platform.

Step 3

In the Azure Portal, click into the Search resources, services and docs textbox and type:

Monitor

Under the results for Services, select Monitor.

Step 4

This will open the Monitor | Overview page.

Note: The Overview page in Monitor allows easy navigation to different types of insights, provided intelligently by Azure automatically analyzing any produced logs. This includes Application, Container, VM and Network insights from the main dashboard.

Step 5

On the left-hand navigation pane, click Logs.

Step 6

This will open the Monitor | Logs page, otherwise known as Log Analytics.

Click the Close (X) button on the Welcome to Log Analytics pane.

Note: Log Analytics in Azure will allow you to simply and effectively query log data generated from an infrastructure.

Step 7

Once this pane has closed, you will be directed to the Select a scope page.

Expand the Resource group dropdown.

Note: The scope page allows you to specify which type of logs you want to review and which resources currently exist as part of your overall Azure subscription. If you had multiple resource groups (or multiple resource groups working in tandem with each other), you could use this drop-down to filter exactly which needs to be queried.

Step 8

Next, expand the Resource types dropdown.

Note: This option, similar to resource groups, allows you to specify a specific type of resource that currently exists in your infrastructure to analyze.

Step 9

Finally, expand the Locations dropdown.

Note: This option is important to consider if you have multiple available resources for specific availability zones. It is key to remember that you can narrow this search further by using all three options in tandem if you wish. For example, you could navigate specifically to a VM that is part of the UK South availability zone and belongs to a specific resource group.

Step 10

At the bottom of the page, click Cancel.

Step 11

This will close the Scope section and bring you to the Query page.

Note: This tool will allow you to run specialized queries allocated by the query scope. There are also options to select a certain time range and save and share log reports. You can also use this page to create and enable automatic alerts, an example of automated logging and alerting. Alert: In the lab environment, the query pane will be greyed out due to not allocating a scope, as currently, no resources exist as part of the infrastructure. Task 1 - Explore Event Viewer

Windows Event Viewer is a native tool for Windows operating systems that allows administrators to capture logs for many different applications and resources being utilized on a Windows device.

In this task, we will access Event Viewer in Windows Server 2019.

Step 1

Connect to PLABDC01.

The Windows Server Manager dashboard will open by default.

Step 2

Click Tools and select Event Viewer.

Step 3

The Event Viewer tool will open.

Step 4

In the left-hand panel, expand Windows Logs and select Application.

Note: Windows logs refer to all types of logs generated by the operating system specifically related to system-level functions. This can be a good diagnostic tool for individual devices.

Step 5

Scroll to the bottom of the page and click the Winlogon event.

The bottom pane will populate to provide more information regarding this event.

Note: This type of event specifies data gathered from logging in to a specific device and can be used for access audit logging for individual devices in a network.

Step 6

Next, on the left-hand panel, navigate to the Applications and Services > Microsoft > Windows > Hyper-V-Compute directory.

Step 7

Click Admin.

From the right-hand panel, click Open.

Step 8

This will open a list of all event logs for the Hyper-V-Compute service, noting that they started successfully.

Note: The Applications and Services directory in event viewer allows you to review logs for specific services and applications. In this example, you investigated Hyper-V, a key application for VM virtualization in private cloud environments.

Keep PLABDC01 in its current state for the next task.

Task 2 - Enable Audit Logging using Local Group Policy Editor

Now that you have seen some of the types of event logs generated automatically by Windows, it is important to ensure that you are capturing information critical to the needs of your infrastructure.

In this task, we will learn how to enable audit logging using Local Group Policy Editor in Windows Server 2019.

Step 1

Ensure you are connected to PLABDC01 with the Event Viewer window open.

Step 2

From the Taskbar, click the Type here to search magnifying glass icon, then type:

Edit Group Policy

From the Best match results, click Edit group policy.

Step 4

This will open the Local Group Policy Editor window.

Note: Group Policy (or Local Group Policy Editor) is another standard Microsoft Management Console (MMC) tool that allows administrators to modify policies applied to Group Policy Objects (GPOs).

Step 5

In the left-hand pane, expand Computer Configuration > Windows Settings > Security Settings > Local Policies.

Then click the Audit Policy directory.

Step 6

Double-click the Audit logon events policy.

Notice that the Security Setting is shown as no auditing.

Step 7

This will open the Audit logon events Properties window.

Tick both the Success and Failure checkboxes.

Click Apply.

Step 8

Audit logging has now been enabled.

Click OK to close the Audit logon events Properties window.

Note: Now that this has been enabled, Event Viewer will start generating security logs noting both success and failure states for logon attempts for the specific device. In private cloud environments, this could be enabled either for all devices on a network or only individual devices.

Step 9

Notice that Audit logon events policy Security Setting is now showing as Success, Failure.

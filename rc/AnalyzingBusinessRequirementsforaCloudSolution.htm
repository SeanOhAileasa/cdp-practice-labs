Exercise 1 - Cloud Requirement Analysis Factors

When building a Cloud infrastructure, business considerations can take a few forms, such as budgetary requirements, software licensing and agreements, and hardware purchasing and logistics (when considering on-premises infrastructure).

Picture a scenario in which a private cloud infrastructure struggles to serve customers due to increased demand on the servers, resulting in poor user experiences. In this case, it could be a business requirement to switch this infrastructure to a public subscription because servers hosted by a Cloud Service Provider (CSP) would be under a Service Level Agreement (SLA) that dictates a certain percentage of uptime, as well as the current infrastructure not meeting user needs. Both are considerations that should be considered before an infrastructure is built.

Task 1 - Calculate a Composite SLA

A Composite SLA is an agreement that spans multiple different products and services offered by Microsoft Azure. They calculate overall performance targets for solutions or workloads involving numerous services, each with different availability levels. For example, consider a web application that uses the App Service and writes to an Azure SQL Database. Currently, the SLAs for these Azure services specify the following uptime guarantees:

  • App Service = 99.95%
  • Azure SQL Database = 99.99%

In this scenario, if either service fails, the whole application fails. So, what is the maximum downtime you would expect for this application?

The individual probability of each service failing is independent, so the guarantees are multiplied to determine the composite SLA for this application as follows:

99.95% × 99.99% = 99.94%

You might be surprised that the composite SLA number is lower than the individual SLAs. However, a solution that relies on multiple services has more potential failure points; thus, the guarantee is lower.

In this task, we will calculate the composite SLA for our web app scenario with an application gateway, a VM, SQL database, and a storage queue.

Task 1 - Explore Private Cloud Penetration Testing

Pacu is mainly designed for security professionals who want to test Amazon Web Services (AWS) security. Pacu runs with the modular architecture that can be used in penetration testing. It contains several tools that can be used for testing security of an AWS environment.

One of the features of Pacu is that it is an open-source application that is freely available on Github, and can work with both MacOS and Linux environments. Pacu also covers the entire penetration testing lifecycle, which can include:

  • Reconnaissance
  • Persistence
  • Privilege escalation
  • Enumeration
  • Data exfiltration
  • Log manipulation
  • Credentials compromise

Pacu contains various modules that can be used in various stages of security testing. Some of the key modules are:

  • confirm_permissions - Lists the permissions for the account being used
  • privesc_scan - Can utilize 20 or more different privilege escalation methods
  • cloudtrail_csv_injection - Can inject malicious formulas in the CSV files that are being exported from CloudTrail
  • disrupt_monitoring - Can disrupt the monitoring capability within the AWS environment
  • backdoor_users - Can be used to establish backdoor account access
  • sysman_ec2_rce - Can break the AWS Simple Systems Manager to gain access to the root account
  • backdoor_ec2_sec_groups - Can be used to add backdoor rules to EC2 security groups

In this task, you will learn to explore Pacu using Kali Linux as an example of a Penetration testing tool used in Cloud environments.
